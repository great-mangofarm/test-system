# 에버온 이슈트래커

QA 테스트와 운영 개발요청을 한곳에서 관리하고, **Jira·그룹웨어와 연동**하는 사내 이슈 관리 도구입니다.
제품(Product) → 묶음(Suite) → 항목(테스트케이스/운영이슈) 구조로, 팀이 테스트 진행도와 배포 단위를 한눈에 추적합니다.

> 운영 도메인: **https://issue.datasystem.app**

---

## 무엇을 하는 앱인가

이 앱은 두 가지 성격의 작업을 같은 틀에서 다룹니다.

1. **QA 테스트케이스 관리** (`qa` 묶음)
   - 기능별 테스트 항목을 등록하고 통과/실패/블로킹/미테스트 상태로 추적
   - 영역별 진행도·통과율을 도넛 차트와 진행바로 시각화
   - 실패한 항목은 처리상태(처리중→처리완료)로 후속 관리

2. **운영이슈 / 개발요청 관리** (`dev` 묶음)
   - 개요·범위/요구사항·기능정의·체크리스트를 갖춘 개발요청서 형태로 이슈 등록
   - 등록 시 **Jira 티켓 자동 생성**, 담당자·우선순위·기한 연동
   - 처리상태 흐름: 미처리 → 처리중 → 개발배포대기 → 개발배포 → 처리완료
   - **배포묶음**으로 "이번 배포에 포함되는 이슈"를 날짜별로 묶어 개발자에게 제시

## 주요 기능

| 기능 | 설명 |
|------|------|
| **제품/묶음 관리** | 제품 아래 여러 묶음(QA/운영), 드래그로 정렬. 관리자 전용 |
| **테스트 진행도** | 묶음별 통과율(QA)·처리완료율(운영) 도넛 + 영역별 진행바 |
| **Jira 연동** | 운영이슈 등록 시 티켓 자동 생성, 담당자/우선순위 매핑, 상태 webhook 동기화 |
| **배포묶음(Deploy Batch)** | 운영이슈를 배포 단위로 그룹핑(이름·예정일·완료상태), 목록을 묶음별 헤더로 표시 |
| **리치텍스트 + 이미지** | Tiptap 에디터, 이미지 붙여넣기·드래그·업로드(Cloudinary) |
| **역할별 노출 제어** | 제품/묶음을 특정 역할에게만 보이도록 설정 |
| **개인 숨김** | 사용자별로 안 보고 싶은 묶음을 숨김(기기별 저장) |
| **사용자/권한 관리** | 관리자가 가입자 역할·팀·Jira 이름 매핑, 비밀번호 초기화 |

---

## 사용자 역할

| 역할 | 권한 |
|------|------|
| **관리자(admin)** | 전체 — 제품/묶음 관리, 이슈 등록·수정, 사용자 관리 |
| **개발자(developer)** | 이슈/테스트케이스 등록·수정·삭제 + Jira 연동. 제품/묶음 구조 관리·사용자 관리는 불가 |
| **스태프(staff)** | 조회 중심 (가입 시 기본 역할) |

권한은 **Firestore 보안 규칙으로 서버에서 강제**되며, 화면 UI는 그에 맞춰 노출/숨김 처리됩니다.

---

## 화면 구성

- **홈** — 좌측 제품 목록, 우측 선택 제품의 묶음 카드(진행도·도넛). 묶음 클릭 시 상세로 이동
- **묶음 상세** — 항목 목록(표). QA는 행 펼침(아코디언), 운영이슈는 사이드 드로어로 상세 편집. 검색·상태·담당자·배포묶음 필터
- **사용자 관리** — 관리자 전용

## 기술 스택

- **프론트엔드**: Vite + React 19 + TypeScript, Tailwind CSS v4 + shadcn/ui(Radix)
- **데이터**: Firebase Firestore (client SDK) / 인증: Firebase Auth
- **서버리스**: Vercel API Routes (`/api/*`) — Jira 연동, Firebase ID 토큰 인증
- **에디터**: Tiptap / **이미지**: Cloudinary 업로드
- **연동**: Jira REST API(ADF), 그룹웨어(다우오피스 전자결재 — 검토 중)
- **배포**: Vercel (GitHub 연동 자동 배포)

## 로컬 실행

```bash
npm install
npm run dev        # 프론트엔드만 (Vite, localhost:5173)
```

`/api/*` 서버리스 함수까지 로컬에서 돌리려면 `vercel dev`가 필요하며,
시크릿(서비스계정·Jira키)이 Vercel **Production/Preview 스코프에만** 있어
API 변경은 **브랜치 push → Vercel 프리뷰**로 테스트하는 것을 권장합니다. (자세한 내용은 `CLAUDE.md`)

## 배포

```bash
git push git@github-new:great-mangofarm/test-system.git main   # → 프로덕션 자동 배포
```
브랜치를 push하면 프리뷰 배포가 생성됩니다.
